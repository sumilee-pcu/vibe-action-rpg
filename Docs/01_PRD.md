# Project Tiny Vanguard — 제품 요구사항 정의서

## 1. 문서 정보

| 항목 | 내용 |
|---|---|
| 제품명 | Project Tiny Vanguard(가제) |
| 문서 버전 | 1.3 |
| 단계 | 구현 진행 중 |
| 제품 형태 | 싱글 플레이 3D 액션 RPG 버티컬 슬라이스 |
| 주요 목적 | 플레이 가능한 데모 제작 + 바이브코딩 교육 사례 구축 |
| 참고 장면 | [Threads 참고 영상](https://www.threads.com/@park_minjun/post/Dalt3d0gEim/media) |

참고 영상은 장르·카메라·전투 밀도·UI 구성의 방향을 파악하는 용도로만 사용한다. 원작의 캐릭터, 명칭, 모델, 사운드, UI 이미지를 복제하지 않고 자체 또는 정식 라이선스 에셋을 사용한다.

## 2. 제품 비전

플레이어가 작은 로우폴리 월드를 이동하며 일반 적과 보스를 상대하고, 짧은 시간 안에 성장과 전투의 완결감을 경험하는 게임을 만든다. 동시에 요구사항 정의, AI 프롬프트 작성, 코드 검토, Unity 에디터 설정, 테스트, 오류 수정, 회고의 전 과정을 교육용 사례로 남긴다.

## 3. 해결하려는 문제

### 플레이어 관점

- 긴 설명 없이도 이동과 전투를 바로 이해할 수 있는 짧은 게임 경험이 필요하다.
- 단순한 기본 공격만 반복하지 않고 회피와 스킬의 선택을 느껴야 한다.
- 10~15분 안에 시작·성장·보스전·종료를 모두 경험해야 한다.

### 교육 관점

- “AI에게 게임을 만들어 달라고 요청”하는 수준을 넘어, 요구사항과 검증 기준에 따라 AI를 통제하는 사례가 필요하다.
- AI가 생성한 코드의 오류, 과도한 구조화, Unity 직렬화 문제 등을 사람이 어떻게 발견하고 고치는지 보여줄 자료가 필요하다.
- 완성된 결과뿐 아니라 실패한 프롬프트와 수정 근거까지 재현 가능하게 기록해야 한다.

## 4. 목표 사용자

### 4.1 최종 플레이어

- 캐주얼한 3D 액션 게임을 즐기는 PC 사용자
- 복잡한 빌드 연구보다 즉각적인 전투를 선호하는 사용자
- 한 세션 10~15분의 짧은 데모를 플레이하려는 사용자

### 4.2 교육자료 학습자

- Unity 에디터 기본 조작을 경험한 학습자
- C# 변수, 조건문, 반복문, 클래스의 기초를 이해하는 학습자
- 생성형 AI를 사용하지만 요구사항 분해·검증 방법을 배우고 싶은 학습자

## 5. 핵심 경험

1. 플레이어는 시작 후 30초 안에 이동과 카메라 조작을 이해한다.
2. 2분 안에 첫 적과 전투하고 공격·회피·피격 피드백을 경험한다.
3. 일반 적을 처치해 경험치를 얻고 한 번 이상 성장한다.
4. 쿨다운이 다른 두 개 이상의 스킬을 상황에 맞게 사용한다.
5. 마지막 보스를 처치하면 명확한 결과 화면을 본다.

## 6. 제품 범위

### 6.1 MVP 범위

- 플레이어 캐릭터 1종
- 작은 전투 맵 1개
- 일반 근접 적 1종
- 정예 또는 보스 적 1종
- 기본 공격 콤보 또는 단일 연속 공격
- 회피 1종
- 액티브 스킬 2종
- 체력, 경험치, 레벨 시스템
- 적 탐지·추적·공격·피격·사망 AI 상태
- HUD, 적 머리 위 정보, 데미지 숫자
- 시작, 승리, 패배, 재시작 흐름
- macOS 실행 빌드

### 6.2 버티컬 슬라이스 확장 범위

- 일반 적 2종과 보스 1종
- 액티브 스킬 총 3종
- 보스 공격 패턴 2개 이상
- 간단한 전투 튜토리얼
- 옵션: 음량, 마우스 감도, 화면 해상도
- Windows 빌드와 입력 검증
- 효과음, 배경음악, 타격 이펙트의 최종 폴리싱

### 6.3 이번 범위에서 제외

- 오픈 월드
- 온라인 멀티플레이
- 절차적 월드 생성
- 대화 선택과 분기형 스토리
- 복잡한 인벤토리·제작·상점
- 다수의 장비와 스탯 빌드
- 모바일 출시
- 자체 3D 캐릭터 모델링 전체 과정
- 생성형 AI가 런타임에서 NPC 행동을 결정하는 기능

## 8. 기능 요구사항

### 8.1 플레이어와 카메라

- **FR-PLAYER-001** 플레이어는 키보드 입력으로 카메라 기준 전후좌우 이동을 할 수 있어야 한다.
- **FR-PLAYER-002** 플레이어는 마우스로 3인칭 카메라를 회전할 수 있어야 한다.
- **FR-PLAYER-003** 이동 방향으로 캐릭터가 자연스럽게 회전해야 한다.
- **FR-PLAYER-004** 공격 중 이동 허용량은 공격 데이터에서 조정할 수 있어야 한다.
- **FR-PLAYER-005** 회피 중에는 짧은 무적 시간이 적용되어야 한다.
- **FR-CAMERA-001** 카메라는 플레이어를 추적하고 장애물에 가려지는 상황을 최소화해야 한다.

### 8.2 전투

- **FR-COMBAT-001** 기본 공격 입력은 공격 애니메이션과 판정을 실행해야 한다.
- **FR-COMBAT-002** 공격 판정은 한 번의 공격에서 동일 대상을 중복 타격하지 않아야 한다.
- **FR-COMBAT-003** 공격 데이터는 피해량, 판정 시점, 경직, 재사용 대기시간을 데이터로 조정할 수 있어야 한다.
- **FR-COMBAT-004** 피격 시 체력 감소, 피격 반응, 데미지 숫자, 효과음 중 적용 가능한 피드백을 동시에 제공해야 한다.
- **FR-COMBAT-005** 체력이 0이 되면 사망 상태로 전환되고 추가 행동을 차단해야 한다.
- **FR-COMBAT-006** 스킬은 자원 또는 재사용 대기시간 조건을 만족할 때만 실행되어야 한다.
- **FR-COMBAT-007** UI는 각 스킬의 남은 재사용 대기시간을 표시해야 한다.

### 8.3 적 AI

- **FR-AI-001** 적은 비전투·추적·공격·피격·사망 상태를 가져야 한다.
- **FR-AI-002** 플레이어가 탐지 범위에 들어오면 추적을 시작해야 한다.
- **FR-AI-003** 공격 범위에 도달하면 이동을 멈추고 공격해야 한다.
- **FR-AI-004** 플레이어가 이탈 범위를 벗어나면 원래 위치로 복귀해야 한다.
- **FR-AI-005** 여러 적이 플레이어 주변 한 지점에 완전히 겹치지 않도록 해야 한다.
- **FR-AI-006** 보스는 체력 또는 시간 조건에 따라 최소 두 가지 공격 패턴을 사용해야 한다.

### 8.4 성장과 진행

- **FR-PROG-001** 적 처치 시 경험치를 획득해야 한다.
- **FR-PROG-002** 경험치가 기준값에 도달하면 레벨이 오르고 최대 체력 또는 공격력이 상승해야 한다.
- **FR-PROG-003** HUD는 현재 레벨, 체력, 경험치 진행도를 표시해야 한다.
- **FR-PROG-004** 보스 처치 시 승리 상태로 전환되어야 한다.
- **FR-PROG-005** 플레이어 사망 시 패배 화면에서 재시작할 수 있어야 한다.

### 8.5 UI와 피드백

- **FR-UI-001** HUD는 체력, 경험치, 스킬 슬롯을 표시해야 한다.
- **FR-UI-002** 적 머리 위에는 이름·레벨·체력 정보를 표시할 수 있어야 한다.
- **FR-UI-003** 적이 플레이어를 인식했을 때 인식 표시를 잠시 보여야 한다.
- **FR-UI-004** 피해량은 월드 공간 숫자로 표시되고 짧은 시간 후 사라져야 한다.
- **FR-UI-005** 승리·패배·일시정지 화면은 게임 입력과 UI 입력을 명확히 분리해야 한다.

### 8.6 설정과 데이터

- **FR-DATA-001** 캐릭터·적·공격·스킬의 주요 수치는 ScriptableObject 또는 동등한 데이터 자산으로 관리해야 한다.
- **FR-DATA-002** 코드 수정 없이 인스펙터에서 체력, 공격력, 속도, 탐지 거리, 쿨다운을 조정할 수 있어야 한다.
- **FR-DATA-003** 씬 참조 누락이 발생하면 가능한 경우 명확한 경고를 출력해야 한다.

## 9. 비기능 요구사항

- **NFR-PERF-001** 기준 개발 장비에서 1920×1080, 중간 품질, 60fps를 목표로 한다.
- **NFR-PERF-002** 동시 활성 일반 적 10개 상황에서도 장시간 프레임 저하가 없어야 한다.
- **NFR-PERF-003** 반복 생성되는 데미지 숫자와 타격 이펙트는 풀링 적용 여부를 측정 후 결정한다.
- **NFR-COMPAT-001** Apple Silicon 네이티브 에디터와 macOS 빌드에서 실행되어야 한다.
- **NFR-COMPAT-002** 구매·무료 에셋은 URP와 Apple Silicon 호환성을 확인해야 한다.
- **NFR-MAINT-001** 런타임 코드와 에디터 전용 코드를 분리해야 한다.
- **NFR-MAINT-002** 핵심 게임 규칙은 가능한 한 MonoBehaviour 생명주기에 과도하게 결합하지 않아야 한다.
- **NFR-TEST-001** 데미지, 체력, 쿨다운, 경험치 계산에는 EditMode 테스트를 작성한다.
- **NFR-TEST-002** 핵심 플레이 흐름에는 수동 PlayMode 검증 시나리오를 유지한다.
- **NFR-DOC-001** 완료된 기능마다 프롬프트 기록, 검증 결과, 학습 포인트를 남긴다.
- **NFR-LICENSE-001** 외부 에셋의 출처, 라이선스, 사용 범위를 기록한다.

## 10. 인수 기준

- **AC-001 이동** 새 게임 시작 후 플레이어가 WASD와 마우스로 맵을 이동할 수 있다.
- **AC-002 전투 진입** 탐지 범위에 들어가면 적에게 인식 표시가 나타나고 적이 플레이어를 추적한다.
- **AC-003 타격** 기본 공격의 시각적 타격 시점과 실제 피해 적용 시점이 체감상 일치한다.
- **AC-004 회피** 회피 무적 시간에 들어온 공격은 체력을 감소시키지 않는다.
- **AC-005 스킬** 사용한 스킬은 즉시 재사용되지 않고 UI에 남은 시간이 표시된다.
- **AC-006 적 사망** 적 체력이 0이 되면 공격과 이동을 멈추고 경험치를 한 번만 지급한다.
- **AC-007 레벨업** 정해진 경험치에 도달하면 레벨과 스탯이 상승하고 HUD가 갱신된다.
- **AC-008 보스전** 보스가 최소 두 패턴을 사용하고, 처치 후 승리 화면이 표시된다.
- **AC-009 패배** 플레이어 사망 후 패배 화면과 재시작 기능이 정상 작동한다.
- **AC-010 성능** 일반 적 10개가 활성화된 대표 전투에서 목표 성능을 측정하고 결과를 기록한다.
- **AC-011 빌드** macOS 빌드가 에디터 외부에서 실행되고 한 판을 끝낼 수 있다.
- **AC-012 교육 기록** 각 마일스톤에 최소 1개의 개발일지와 주요 프롬프트 기록이 연결되어 있다.

## 11. 기술 방향

| 영역 | 기본 선택 | 비고 |
|---|---|---|
| 엔진 | Unity 6 | 프로젝트 생성 시 정확한 버전 고정 |
| 렌더링 | URP | 에셋 호환성과 수업 재현성 우선 |
| 입력 | Input System | 키보드·마우스, 이후 게임패드 확장 |
| 카메라 | Cinemachine | 구체 버전은 패키지 잠금 파일에 기록 |
| AI 이동 | AI Navigation/NavMesh | 작은 맵과 근접 AI에 적합 |
| 데이터 | ScriptableObject | 스킬·공격·캐릭터 수치 분리 |
| UI | uGUI 우선 | 월드 공간 UI와 교육 난이도 고려 |
| 테스트 | Unity Test Framework | EditMode 중심 + 수동 PlayMode 시나리오 |
| 버전 관리 | Git + Git LFS | 대형 바이너리만 LFS 사용 |
| 1차 개발 | Apple Silicon 개발기 | macOS 네이티브 검증 |
| 교차 검증 | Windows 개발기 | Windows 입력·빌드 확인 |

패키지 버전은 Unity 프로젝트를 생성할 때 공식 지원 상태를 확인한 뒤 `Packages/manifest.json`과 문서에 고정한다.

## 12. 분석 이벤트와 개발 측정값

외부 분석 SDK는 MVP에 포함하지 않는다. 대신 개발용 로그 또는 결과 화면으로 아래 값을 확인할 수 있게 한다.

- 한 판 완료 시간
- 받은 피해량과 사망 횟수
- 기본 공격 및 스킬 사용 횟수
- 적 종류별 처치 수
- 보스전 진입·승리 여부
- 대표 전투의 평균·최저 FPS

## 13. 주요 위험과 대응

| 위험 | 영향 | 대응 |
|---|---|---|
| 범위가 RPG 전체로 확대됨 | 완성 지연 | MVP 제외 목록을 유지하고 변경은 ADR로 기록 |
| AI가 과도한 아키텍처를 생성 | 이해·수정 곤란 | 한 기능당 작은 프롬프트, 코드 리뷰 체크리스트 적용 |
| 애니메이션과 판정 불일치 | 타격감 저하 | 타임라인을 프레임 단위로 측정하고 판정 시점을 데이터화 |
| 에셋 간 스켈레톤·URP 불일치 | 통합 지연 | 구매 전 데모 프로젝트로 호환성 검증 |
| NavMesh 적 겹침 | 전투 가독성 저하 | 정지 거리·회피 반경·공격 슬롯을 단계적으로 적용 |
| macOS에서 플러그인 미지원 | 빌드 실패 | Apple Silicon 및 macOS 지원 명시된 에셋만 채택 |
| 문서가 개발을 따라가지 못함 | 교육 가치 상실 | 완료 정의에 기록 작업 포함, 템플릿 사용 |

## 14. 성공 지표

### 제품 지표

- 처음 플레이하는 사용자가 설명 없이 2분 안에 첫 적을 처치한다.
- 테스트 플레이어의 70% 이상이 15분 안에 보스를 만난다.
- 치명적인 진행 차단 버그 없이 새 게임부터 결과 화면까지 완료한다.
- 대표 전투에서 목표 성능 측정 결과를 만족하거나 미달 원인을 문서화한다.

### 교육 지표

- 각 핵심 기능에 요구사항 → 프롬프트 → 코드 → 테스트 → 회고의 연결이 존재한다.
- 성공 사례뿐 아니라 최소 5개의 실패·수정 사례를 확보한다.
- 학습자가 제공된 단계와 검증 기준만으로 MVP의 핵심 루프를 재현할 수 있다.
- 수업 후 학습자가 AI 결과를 그대로 수용하지 않고 최소 한 가지 결함을 설명할 수 있다.

## 15. 미결정 사항

- 최종 게임명과 세계관
- 사용할 캐릭터·환경·애니메이션 에셋
- 기본 공격을 단일 공격으로 할지 3연속 콤보로 할지
- 타깃 고정 기능을 MVP에 포함할지
- 게임패드 지원을 버티컬 슬라이스에 포함할지
- 교육 과정의 최종 시수와 학습자 선수 수준

미결정 사항은 실제 구현을 막지 않는 한 가정으로 진행하고, 범위나 구조에 큰 영향을 주는 결정만 ADR로 확정한다.
