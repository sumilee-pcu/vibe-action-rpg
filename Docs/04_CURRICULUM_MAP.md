# 바이브코딩 교육과정 구성안

기본안은 3시간 × 8회다. 실제 학습자 수준에 따라 각 회차를 2~4시간으로 조절할 수 있다.

| 회차 | 결과물 | Unity·C# 개념 | 바이브코딩 역량 | 검증 과제 |
|---:|---|---|---|---|
| 1 | PRD와 빈 프로젝트 | 프로젝트 구조, 컴포넌트 | 요구사항을 작은 작업으로 분해 | 빈 빌드와 버전 고정 |
| 2 | 이동·카메라 | Input System, Transform, CharacterController | 환경 정보를 포함한 프롬프트 | 프레임 독립 이동 검사 |
| 3 | 체력·기본 공격 | 인터페이스, 이벤트, 충돌 | 인수 기준으로 코드 요청 | 중복 타격 테스트 |
| 4 | 적 상태 머신 | NavMesh, 상태 전이 | AI 설계안 비교·비평 | 추적·공격·복귀 시나리오 |
| 5 | 회피·스킬 | 코루틴/시간, ScriptableObject | 경계 조건 명시 | 무적·쿨다운 테스트 |
| 6 | UI·성장 | uGUI, 이벤트 구독, 데이터 흐름 | 콘솔 오류 기반 디버깅 | 경험치 1회 지급 검사 |
| 7 | 보스·게임 흐름 | 패턴, 상태 관리, 씬 흐름 | 큰 요구를 단계별 프롬프트로 분리 | 승리·패배 E2E 검사 |
| 8 | 최적화·빌드·회고 | Profiler, 빌드, 플랫폼 차이 | AI 결과 평가와 문서화 | 성능·빌드 보고서 |

## 공통 학습 성과

학습자는 과정 종료 후 다음을 할 수 있어야 한다.

1. 자연어 아이디어를 기능 요구사항과 인수 기준으로 바꾼다.
2. AI가 읽어야 할 코드와 수정 범위를 제한한다.
3. 생성된 Unity C# 코드의 대표적인 결함을 찾는다.
4. Unity 콘솔, 테스트, 플레이 시나리오로 결과를 검증한다.
5. 실패한 프롬프트를 근거를 들어 개선한다.
6. 코드·프롬프트·검증·회고를 연결된 개발 기록으로 남긴다.

## 평가 비율 예시

| 평가 항목 | 비율 |
|---|---:|
| 요구사항·프롬프트 품질 | 20% |
| 기능 구현 | 25% |
| 테스트와 검증 증거 | 25% |
| AI 결과 비평·수정 | 20% |
| 문서화·회고 | 10% |

게임의 화려함보다 요구사항 충족과 검증 능력을 더 높게 평가한다.
