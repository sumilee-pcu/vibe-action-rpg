# 바이브코딩 기록 및 교육자료화 절차

## 1. 목적

이 절차는 “AI가 코드를 만들어 준 과정”이 아니라, 사람이 목표와 검증 기준을 세우고 AI의 결과를 평가·수정한 과정을 남기기 위한 것이다. 모든 기능은 작은 실험 단위로 진행하며, 한 번의 대형 프롬프트로 전체 게임을 생성하지 않는다.

## 2. 기능 하나를 만드는 표준 사이클

### 2.1 요구사항 선택

1. [[01_PRD]]에서 기능 요구사항 ID 하나 또는 밀접한 2~3개를 선택한다.
2. 플레이 모드에서 관찰할 수 있는 완료 조건을 한 문장으로 쓴다.
3. 이번 작업에 포함하지 않을 것을 적는다.
4. 수정될 가능성이 높은 파일·프리팹·씬을 확인한다.

예시:

```text
대상: FR-COMBAT-002
완료 조건: 한 번의 검 휘두르기로 동일 적에게 피해가 한 번만 들어간다.
제외: 콤보, 치명타, 속성 피해
검증: 공격 판정 안에 적 콜라이더가 두 개 있어도 체력 감소는 한 번이다.
```

### 2.2 기준 상태 저장

작업 전 다음을 기록한다.

- 실행되는 Unity 버전
- 현재 Git 브랜치와 커밋
- 콘솔 오류 유무
- 재현 또는 구현 전 상태의 스크린샷·짧은 영상
- 관련 씬과 테스트 방법

### 2.3 첫 프롬프트 작성

[[Templates/PROMPT_LOG_TEMPLATE]]을 복제해 다음 내용을 포함한다.

- 역할과 기술 환경
- 수정 범위
- 요구사항 ID
- 인수 기준
- 기존 구조와 제약
- 금지 사항
- 기대하는 산출물
- 실행할 테스트

좋은 프롬프트는 구현 방법을 모두 대신 설계하지 않는다. AI가 먼저 관련 코드를 읽고 변경 계획을 짧게 제시하게 한 뒤, 작은 범위에서 구현하도록 한다.

### 2.4 AI 결과 검토

결과를 적용하기 전 최소한 다음을 확인한다.

- 존재하지 않는 Unity API나 패키지를 가정하지 않았는가?
- `Update()`에서 불필요한 검색·할당을 반복하지 않는가?
- 직렬화 필드와 씬 참조가 깨지지 않는가?
- 동일 이벤트가 여러 번 등록되지 않는가?
- 사망·일시정지·씬 전환 같은 경계 상태를 처리하는가?
- 기능 범위를 넘어 대규모 리팩터링하지 않았는가?
- 사용한 에셋과 코드의 라이선스가 명확한가?

### 2.5 작은 단위로 적용

1. 코드 컴파일이 되는 최소 변경을 적용한다.
2. Unity 콘솔 오류를 0개로 만든다.
3. 자동 테스트가 있으면 먼저 실행한다.
4. 최소 테스트 씬에서 확인한다.
5. 통합 씬에서 회귀를 확인한다.
6. 한 커밋에는 하나의 설명 가능한 의도를 담는다.

### 2.6 검증 기록

통과 여부만 적지 말고 증거를 남긴다.

| 검증 종류 | 기록할 증거 |
|---|---|
| 컴파일 | Unity 콘솔 오류 0개 |
| EditMode | 테스트 이름, 통과 개수, 실행 시각 |
| PlayMode | 조작 순서, 기대 결과, 실제 결과 |
| 시각 | 스크린샷 또는 짧은 영상 경로 |
| 성능 | 테스트 씬, 해상도, 적 수, 평균·최저 FPS |
| 빌드 | 플랫폼, 빌드 설정, 실행 성공 여부 |

### 2.7 회고와 교육 포인트 추출

[[Templates/DEVLOG_TEMPLATE]]에 다음을 적는다.

- AI가 잘 이해한 부분
- AI가 틀렸거나 과도하게 만든 부분
- 사람이 추가로 판단한 부분
- 프롬프트를 다시 쓴다면 바꿀 점
- 학습자가 흔히 빠질 함정
- 수업에서 던질 질문

## 3. 프롬프트 버전 관리 규칙

- 파일명: `YYYY-MM-DD_FEATURE_SHORT_NAME_vNN.md`
- 원문 프롬프트를 수정해 덮어쓰지 않는다.
- 후속 프롬프트는 새 버전으로 만들고 이전 버전과 차이를 적는다.
- AI 답변 전체를 무조건 저장하지 않는다. 핵심 제안, 생성 파일, 잘못된 가정, 채택·기각 이유를 기록한다.
- API 키, 토큰, 개인 경로 등 비밀 정보는 문서에 포함하지 않는다.

## 4. 개발일지 규칙

- 파일명: `YYYY-MM-DD_MILESTONE_TOPIC.md`
- 하루에 여러 기능을 다뤄도 학습 목표가 다르면 문서를 분리한다.
- 관련 요구사항, 프롬프트, 커밋, 테스트, 오류 문서를 위키 링크로 연결한다.
- “성공함” 대신 재현 가능한 조작과 관찰 결과를 쓴다.

## 5. 오류 지식베이스 규칙

오류는 해결된 뒤에만 정리하는 것이 아니라, 재현되는 순간 최소 정보를 남긴다.

1. 화면에 보이는 증상
2. 정확한 오류 메시지
3. 발생 조건과 최소 재현 단계
4. 시도했지만 실패한 방법
5. 근본 원인
6. 최종 해결 방법
7. 회귀 테스트
8. 일반화된 학습 포인트

## 6. 교육자료로 변환하는 방법

각 마일스톤이 끝나면 기록을 다음 구조로 변환한다.

1. **도입 문제**: 플레이 화면에서 무엇이 부족한지 보여준다.
2. **개념**: Unity·C# 개념을 10분 이내로 설명한다.
3. **요구사항 읽기**: 관련 FR/AC를 학습자가 해석한다.
4. **프롬프트 설계**: 나쁜 프롬프트를 개선하는 활동을 한다.
5. **AI 결과 비평**: 일부러 결함이 있는 결과를 찾아낸다.
6. **실습**: 작은 기능을 구현한다.
7. **검증**: 체크리스트와 테스트로 완료를 증명한다.
8. **확장 과제**: 데이터나 규칙을 바꿔 다른 결과를 만든다.
9. **회고**: AI와 사람의 역할을 구분해 설명한다.

## 7. 커밋 추적 형식

권장 커밋 제목:

```text
feat(combat): prevent duplicate hit per swing [FR-COMBAT-002]
```

권장 본문:

```text
Prompt: Docs/PromptLog/2026-07-XX_duplicate_hit_v01.md
Test: DamageReceiverTests, manual scenario COMBAT-002-A
DevLog: Docs/DevLog/2026-07-XX_M2_duplicate-hit.md
```

## 8. 품질 게이트

기능을 완료 처리하기 전에 [[Templates/FEATURE_DONE_CHECKLIST]]을 통과해야 한다. 특히 AI가 생성한 코드에는 “작동함”과 “설명·수정 가능함”을 별도 항목으로 평가한다.
