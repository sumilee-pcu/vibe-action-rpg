## Context

이 프로젝트는 새 Unity 6.3 LTS URP 프로젝트로 시작하며 기존 런타임 코드는 없다. 첫 결과물은 10~15분 분량의 싱글 플레이 3인칭 액션 RPG 버티컬 슬라이스이고, 동일한 개발 이력을 바이브코딩 교육자료로 재사용한다. 1차 개발과 빌드는 Apple Silicon macOS에서 수행하고 Windows에서 교차 검증한다.

주요 제약은 작은 1인 개발 범위, AI 생성 코드의 설명 가능성, Unity 씬·프리팹 직렬화 안전성, URP 에셋 호환성, 코드·프롬프트·테스트·회고의 추적 가능성이다.

## Goals / Non-Goals

**Goals:**

- 플레이어 이동부터 보스 처치 또는 플레이어 사망까지 완결된 한 판을 제공한다.
- 게임 수치와 런타임 상태를 분리해 인스펙터 조정과 자동 테스트를 가능하게 한다.
- 기능을 작은 샌드박스 씬에서 검증한 뒤 통합 씬으로 옮길 수 있게 한다.
- 요구사항 ID를 프롬프트, 커밋, 테스트와 교육 문서의 공통 추적 키로 사용한다.
- macOS와 Windows에서 동일한 핵심 게임플레이를 제공한다.

**Non-Goals:**

- 온라인 멀티플레이, 오픈 월드, 복잡한 인벤토리·제작·상점은 구현하지 않는다.
- 초기부터 ECS, 의존성 주입 프레임워크, 비헤이비어 트리를 도입하지 않는다.
- 모든 에셋을 직접 모델링하거나 런타임 생성형 AI를 사용하지 않는다.
- 측정 없이 모든 오브젝트에 풀링과 조기 최적화를 적용하지 않는다.

## Decisions

### Unity 6.3 LTS, Universal 3D, URP 17.3 사용

프로젝트는 설치된 Unity 6000.3.11f1의 Universal 3D 템플릿으로 만들고 패키지 버전을 잠근다. HDRP는 목표 화면에 비해 무겁고 교육 환경 재현 범위를 좁히며, Built-in 파이프라인은 새 교육 프로젝트의 장기성을 떨어뜨리므로 채택하지 않는다.

### 공통 규칙과 Unity 어댑터 분리

체력, 피해, 경험치, 레벨과 쿨다운 계산은 가능한 한 순수 C# 규칙으로 두고 MonoBehaviour는 입력, 애니메이션, 물리, 씬 참조를 연결한다. 이 방식은 EditMode 테스트와 교육용 설명을 쉽게 한다. 모든 시스템을 완전한 클린 아키텍처로 분해하는 방식은 현재 범위에 과도해 채택하지 않는다.

### ScriptableObject에는 정의 데이터만 저장

`ActorDefinition`, `AttackDefinition`, `SkillDefinition`, `EnemyDefinition`은 조정 가능한 정의값을 보관한다. 현재 체력, 남은 쿨다운, 이미 타격한 대상 같은 런타임 상태는 인스턴스가 소유한다. ScriptableObject에 런타임 상태를 저장하면 Play Mode 종료 후 상태 혼란과 인스턴스 간 공유 버그가 발생할 수 있어 금지한다.

### 단순 명시적 적 상태 머신

일반 적은 Idle, Chase, Attack, Hit, Return, Dead 상태를 명시적으로 갖는다. 초기에는 열거형과 작은 상태 전이 함수로 구현한다. 상태 패턴이나 비헤이비어 트리는 적 종류와 전이 복잡도가 실제로 증가한 뒤 검토한다.

### 공격 단위 중복 타격 차단

각 공격 실행은 고유한 공격 컨텍스트와 이미 타격한 대상 집합을 갖는다. 적에게 여러 콜라이더가 있어도 한 번의 공격 실행에서 피해는 한 번만 적용한다. 단순 콜라이더 비활성 방식은 연속 공격과 여러 공격자가 섞일 때 오류가 생기기 쉬워 채택하지 않는다.

### 이벤트 기반 UI 갱신

HUD는 체력, 경험치, 레벨, 쿨다운 변화 이벤트에 반응한다. 매 프레임 전체 게임 상태를 조회하는 방식은 불필요한 결합과 비용을 만들고 교육적으로 데이터 흐름을 숨기므로 사용하지 않는다.

### Markdown을 교육 문서 원본으로 사용

`Docs`의 Obsidian 호환 Markdown을 문서 원본으로 두고 요구사항 ID로 OpenSpec과 연결한다. Notion은 교육과정이 확정된 후 공유용 파생본으로 사용할 수 있다. 두 위치를 동시에 원본으로 운영하지 않는다.

## Risks / Trade-offs

- [에셋의 URP·Apple Silicon 비호환] → 도입 전 별도 씬에서 셰이더, 애니메이션, 빌드를 검증하고 라이선스와 버전을 기록한다.
- [AI가 과도한 구조를 생성] → 한 작업에 1~3개 요구사항만 포함하고 완료 체크리스트에서 설명 가능성을 검토한다.
- [애니메이션과 공격 판정 시점 불일치] → 판정 시작·종료를 데이터화하고 프레임 단위 플레이 검증 증거를 남긴다.
- [NavMeshAgent가 한 지점에 겹침] → 정지 거리, 회피 반경, 공격 슬롯 순서로 단순한 대응부터 적용한다.
- [이벤트 구독 누수] → OnEnable/OnDisable 대칭과 사망·씬 재시작 회귀 시나리오를 유지한다.
- [교육 문서 작업으로 구현 일정 증가] → 기록을 완료 정의에 포함하고 템플릿으로 작성 시간을 제한한다.
- [macOS 결과가 Windows와 다름] → 각 마일스톤 완료 시점이 아니라 M6 이전부터 대표 빌드로 입력과 셰이더를 교차 확인한다.

## Migration Plan

1. Unity 6000.3.11f1 Universal 3D 프로젝트를 `UnityProject`에 생성한다.
2. 패키지와 프로젝트 설정을 고정하고 빈 macOS 빌드를 확인한다.
3. `CombatSandbox`에서 플레이어, 전투, 적 AI, 성장 순으로 기능을 추가한다.
4. 검증된 기능을 `VerticalSlice` 씬에 통합한다.
5. 보스와 게임 세션 흐름을 추가하고 macOS 전체 빌드를 검증한다.
6. Windows에서 입력, 렌더링과 한 판 흐름을 교차 검증한다.

코드 마이그레이션은 없으며 각 기능은 작은 커밋으로 되돌릴 수 있게 한다. 에디터 또는 패키지 업그레이드가 필요하면 별도 OpenSpec 변경으로 다룬다.

## Open Questions

- 기본 공격을 MVP에서 단일 연속 공격으로 할지 3단 콤보로 할지 결정해야 한다.
- 타깃 고정과 게임패드 입력을 버티컬 슬라이스 범위에 포함할지 결정해야 한다.
- 사용할 캐릭터, 애니메이션, 환경 에셋과 라이선스를 선정해야 한다.
- 교육과정의 실제 회차당 시간과 학습자 선수 지식 수준을 확정해야 한다.
